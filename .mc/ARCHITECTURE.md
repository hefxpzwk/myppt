# ARCHITECTURE.md

이 문서는 프로젝트의 기술 구조, 설계 의도, 시스템 경계, 폴더 방향, 데이터 흐름을 정의한다.
새로운 AI 에이전트나 개발자가 이 문서를 읽고 나면, 왜 이 프로젝트가 존재하는지와 어떤 방식으로 확장해야 하는지를 이해할 수 있어야 한다.

---

# 1. 프로젝트 개요

## 1.1 목적

이 프로젝트는 HTML 기반 발표 자료를 한곳에서 관리하고 실행하기 위한 정적 프론트엔드 허브를 구축하는 것을 목표로 한다.

기존 문제는 아래와 같다.

- 발표마다 HTML 기반 자료를 따로 생성한다.
- 자료가 쌓일수록 관리 위치와 실행 방식이 흩어진다.
- 새 발표를 만들 때마다 프롬프트, 스킬, 개인 규칙을 반복 입력해야 한다.

이 프로젝트는 그 문제를 다음 방식으로 해결한다.

- 발표 자료를 하나의 웹 허브에서 탐색한다.
- 발표 메타데이터를 일관된 구조로 관리한다.
- 배포된 웹에서 바로 발표를 시작할 수 있게 한다.
- GitHub와 Vercel을 이용해 서버 없이 운영한다.

## 1.2 핵심 개념

이 프로젝트는 크게 세 가지 자산으로 구성된다.

1. 대시보드 UI
2. 발표 메타데이터
3. 실제 HTML 발표 파일

대시보드는 메타데이터를 보여주고, 메타데이터는 실제 발표 파일의 경로를 연결한다.

---

# 2. 시스템 성격

## 2.1 아키텍처 원칙

이 시스템은 아래 성격을 가진다.

- Static Frontend Only
- Git 기반 콘텐츠 관리
- Vercel 자동 배포
- 런타임 서버/API 없음

즉, 운영 중에 별도 서버 연산이 필요하지 않으며, 배포 결과물은 정적 파일 세트여야 한다.

## 2.2 비목표

아래 항목은 현재 아키텍처의 목표가 아니다.

- 백엔드 서버 구축
- DB 기반 발표 관리
- 로그인/권한 시스템 도입
- CMS 구축
- 실시간 협업 편집 기능

이 비목표는 아키텍처 범위를 지키기 위한 기준이다.

---

# 3. 핵심 사용자 흐름

## 3.1 메인 플로우

기본 사용자 흐름은 아래와 같다.

1. 사용자가 서비스 루트로 접속한다.
2. 대시보드에서 발표 목록을 본다.
3. 제목, 태그, 정렬 기준 등으로 발표를 찾는다.
4. 발표를 선택한다.
5. 발표 전용 뷰로 이동한다.
6. 발표 콘텐츠를 로드한다.
7. 전체화면 또는 일반 모드로 발표를 진행한다.

## 3.2 실패 플로우

아래 실패 상황을 고려해야 한다.

- 발표 경로가 잘못된 경우
- 메타데이터는 있으나 실제 파일이 없는 경우
- URL 파라미터가 잘못된 경우
- 정적 배포 경로가 예상과 다른 경우

이 경우 대시보드로 복귀할 수 있는 안전한 UI가 필요하다.

# 4. 기술 구조

## 4.1 권장 스택

구체적인 구현 프레임워크는 변경 가능하지만, 아래 원칙을 만족해야 한다.

- 정적 빌드가 가능해야 한다.
- Vercel에 무리 없이 배포 가능해야 한다.
- 발표 HTML 파일을 그대로 서빙할 수 있어야 한다.
- 라우팅과 정적 자산 경로를 명확히 제어할 수 있어야 한다.

초기 구현 후보는 아래 범주 중 하나를 상정한다.

- Vite 기반 SPA
- Next.js 정적 export 기반 앱
- 단순 정적 사이트 생성기

단, 어떤 선택이든 서버리스 정적 배포 원칙을 유지해야 한다.

## 4.2 배포 모델

배포 흐름은 아래와 같다.

1. 로컬에서 발표 HTML 생성
2. 저장소에 발표 파일 및 메타데이터 반영
3. GitHub에 커밋 및 푸시
4. Vercel이 최신 커밋을 자동 배포
5. 배포된 웹에서 즉시 발표 가능

## 4.3 운영 기준 브랜치

- `main`은 운영 배포 기준 브랜치다.
- `develop`은 개발 통합 브랜치다.
- 일반 변경은 작업 브랜치에서 시작해 `develop`을 거친다.

브랜치 상세 규칙은 `CONVENTION.md`를 따른다.

# 5. 디렉터리 구조 설계

## 5.1 권장 디렉터리

프로젝트는 아래 구조를 기본안으로 사용한다.

```text
/
├─ AGENT.md
├─ .mc/
│  ├─ RULES.md
│  ├─ CONVENTION.md
│  ├─ ARCHITECTURE.md
│  └─ skills/
├─ src/
│  ├─ pages/
│  │  ├─ dashboard/
│  │  └─ presentation/
│  ├─ components/
│  ├─ data/
│  ├─ utils/
│  └─ styles/
├─ public/
│  └─ slides/
└─ ...
```

## 5.2 각 영역의 책임

- `.mc/skills`
  - 반복 작업을 줄이기 위한 프로젝트 전용 실행 가이드
- `src/pages/dashboard`
  - 발표 목록, 검색, 필터, 정렬, 진입 UI
- `src/pages/presentation`
  - 발표 로더, 발표 실행 UI, 전체화면 제어
- `src/components`
  - 여러 페이지에서 공유하는 UI 컴포넌트
- `src/data`
  - 발표 메타데이터와 관련 타입
- `src/utils`
  - 경로 해석, 포맷 변환, 유효성 검사 등의 보조 로직
- `src/styles`
  - 전역 스타일, 토큰, 테마
- `public/slides`
  - 실제 HTML 발표 파일 및 발표별 정적 자산

## 5.3 구조 설계 이유

이 구조는 아래 이유로 채택한다.

- 대시보드와 발표 실행 책임을 분리할 수 있다.
- 발표 콘텐츠와 앱 코드를 분리할 수 있다.
- 정적 배포 시 발표 HTML을 그대로 서빙할 수 있다.
- 발표 메타데이터를 독립적으로 다룰 수 있다.

---

# 6. 데이터 구조 설계

## 6.1 발표 메타데이터

발표 목록은 정적 데이터 파일로 관리한다.
기본 운영 방식은 `public/slides`의 HTML 파일을 스캔해 메타데이터를 빌드/개발 시작 전에 자동 생성하는 방식이다.

후보 형식:

- `slides.ts`
- `slides.json`

현재 권장 구현:

- 생성 스크립트: `scripts/generate-presentations.mjs`
  - 발표 HTML을 스캔해 메타데이터를 만들고, 각 발표 첫 화면 PNG를 `public/slides/.thumbnails`에 생성
- 생성 결과: `src/data/slides.generated.ts`
- 앱 진입 데이터: `src/data/slides.ts`

권장 최소 필드는 아래와 같다.

- `id`
- `title`
- `path`
- `updatedAt`

권장 확장 필드는 아래와 같다.

- `description`
- `tags`
- `thumbnail`
- `author`
- `featured`

## 6.2 메타데이터 예시

```json
{
  "id": "agent-workflow-demo",
  "title": "Agent Workflow Demo",
  "path": "/slides/agent-workflow-demo/index.html",
  "updatedAt": "2026-04-03",
  "description": "에이전트 기반 발표 제작 흐름 소개",
  "tags": ["agent", "workflow"],
  "thumbnail": "/slides/agent-workflow-demo/thumbnail.png"
}
```

## 6.3 데이터 설계 원칙

- `id`는 URL과 내부 식별에 재사용 가능해야 한다.
- `path`는 실제 배포 결과물과 1:1로 대응해야 한다.
- 필수 필드는 적게 유지하고, 선택 필드는 점진적으로 늘린다.
- 데이터 포맷을 바꿀 때는 기존 발표 링크의 호환성을 먼저 검토한다.

# 7. 발표 렌더링 구조

## 7.1 렌더링 방식

발표 뷰는 실제 HTML 발표 파일을 로드하는 역할을 가진다.

가능한 방식은 아래 둘 중 하나다.

1. 별도 라우트에서 직접 발표 HTML로 이동
2. 앱 내부 발표 뷰에서 `iframe`으로 임베드

현재 권장 방향은 앱 차원의 UI 제어가 필요한 경우 `iframe` 기반 발표 뷰를 우선 검토하는 것이다.

## 7.2 `iframe` 사용 시 고려사항

- 전체화면 제어가 가능해야 한다.
- 발표 경로 오류를 감지할 수 있어야 한다.
- 외부 출처가 아닌 저장소 내부 정적 파일만 대상으로 한다.
- 필요 시 보안 속성과 스타일 격리를 고려한다.

## 7.3 직접 이동 방식 고려사항

- 발표 HTML 자체가 완전한 독립 앱일 때 적합하다.
- 대신 허브 UI와 발표 UI의 연결이 느슨해질 수 있다.
- 뒤로가기 경험과 네비게이션 설계를 따로 고려해야 한다.

---

# 8. 라우팅 원칙

## 8.1 페이지 책임 분리

라우팅은 최소한 아래 두 영역을 분리해야 한다.

- 대시보드 라우트
- 발표 라우트

## 8.2 경로 설계 원칙

- 배포 환경에서 깨지지 않는 경로를 사용한다.
- 발표 파일 경로와 앱 라우트를 혼동하지 않는다.
- base path가 생겨도 상대 경로 전략이 유지되도록 설계한다.

## 8.3 오류 대응

아래 상황에서는 사용자 친화적 대응이 필요하다.

- 존재하지 않는 발표 `id`
- 누락된 `path`
- 발표 파일 로드 실패

기본 대응 방향:

- 오류 메시지 표시
- 대시보드 복귀 버튼 제공
- 가능한 경우 문제 경로를 로그나 상태로 확인 가능하게 유지

# 9. 성능 및 유지보수 고려사항

## 9.1 성능 원칙

- 초기 대시보드 로딩은 빠르게 유지한다.
- 발표가 많아져도 메타데이터 탐색이 무겁지 않도록 한다.
- 큰 썸네일이나 대용량 정적 자산은 필요 시 최적화한다.

## 9.2 유지보수 원칙

- 새 발표 추가 절차가 단순해야 한다.
- 메타데이터 수정만으로 대시보드 반영이 가능해야 한다.
- 다음 에이전트가 구조를 빠르게 이해할 수 있도록 문서를 유지한다.

---

# 10. 보안 및 안정성 고려사항

## 10.1 정적 앱 보안 관점

서버가 없더라도 아래는 고려해야 한다.

- 경로 조작을 통한 의도치 않은 파일 접근 방지
- 사용자 입력을 DOM에 삽입할 때 XSS 방지
- 외부 스크립트 의존 최소화
- 발표 파일 임베드 시 출처와 권한 범위 확인

## 10.2 운영 안정성

- 잘못된 발표 하나가 대시보드 전체를 깨뜨리지 않아야 한다.
- 메타데이터 일부 오류가 전체 앱 실패로 번지지 않아야 한다.
- 잘못된 링크가 있어도 사용자는 안전하게 복귀할 수 있어야 한다.

---

# 11. 확장 방향

향후 아래 기능은 확장 후보가 될 수 있다.

- 최근 본 발표
- 즐겨찾기
- 태그 필터 고도화
- 썸네일 자동 생성
- 발표 템플릿 카탈로그
- 발표 검색 개선

단, 아래 원칙은 유지한다.

- 서버 없이 동작할 것
- 정적 배포 가능할 것
- Git 중심 운영 모델을 깨지 않을 것

---

# 12. 최종 원칙

이 프로젝트의 핵심은 화려한 플랫폼이 아니라 발표 운영 마찰을 줄이는 것이다.

구조 판단이 필요할 때는 아래 질문을 기준으로 삼는다.

- 이 변경이 발표 추가를 더 쉽게 만드는가
- 이 변경이 정적 배포 모델을 유지하는가
- 이 변경이 다음 에이전트의 이해 비용을 줄이는가

셋 중 하나라도 명확히 충족하지 못하면 구조 변경은 재검토한다.
