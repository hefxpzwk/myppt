# CONVENTION.md

이 문서는 이 프로젝트에서 사용하는 코드, 문서, Git 협업, 브랜치, 커밋, 검증 규칙을 정의한다.
목표는 여러 AI 에이전트와 사람이 동일한 스타일과 절차로 작업하도록 만드는 것이다.

이 프로젝트는 정적 프론트엔드 발표 허브이므로, 컨벤션 역시 일반 웹앱 규칙이 아니라 이 저장소의 운영 방식에 맞게 적용한다.

---

# 1. 공통 원칙

## 1.1 우선순위

작업 시 우선순위는 아래와 같다.

1. 안전성
2. 정확성
3. 일관성
4. 가독성
5. 확장성
6. 간결성

## 1.2 기본 철학

- 오래 유지 가능한 구조를 우선한다.
- 숨겨진 규칙보다 명시적 규칙을 우선한다.
- 과한 추상화보다 읽히는 구조를 우선한다.
- 시각적으로 예쁜 코드보다 수정 가능한 코드를 우선한다.
- 발표 자료 추가와 운영이 쉬운 방향을 우선한다.

## 1.3 금지 사항

- 의미 없는 축약어 사용
- 이유 없는 디렉터리 분리
- 매직 문자열 방치
- 문서와 실제 동작의 불일치 방치
- 한 작업에서 서로 다른 목적의 변경을 섞는 것

# 2. Git 및 브랜치 규칙

## 2.1 브랜치 전략

이 프로젝트의 기본 브랜치 전략은 아래와 같다.

- `main`
  - 운영 배포 브랜치
  - Vercel이 참조하는 기준 브랜치
- `develop`
  - 개발 통합 브랜치
  - 기능 개발이 먼저 모이는 브랜치

일반 기능 작업은 반드시 `develop`에서 분기한 작업 브랜치에서 수행한다.

## 2.2 작업 브랜치 네이밍

작업 브랜치는 아래 형식을 따른다.

- `feat/<short-description>`
- `fix/<short-description>`
- `docs/<short-description>`
- `chore/<short-description>`
- `refactor/<short-description>`
- `hotfix/<short-description>`

규칙:

- 모두 소문자 사용
- 단어 구분은 하이픈 사용
- 너무 추상적인 이름 금지

좋은 예:

- `feat/presentation-dashboard`
- `fix/slide-path-resolution`
- `docs/update-architecture`

나쁜 예:

- `feature1`
- `my-branch`
- `test`

## 2.3 브랜치 운용 규칙

- `main`에서 직접 기능 개발하지 않는다.
- `develop`에서 직접 장기 작업하지 않는다.
- 파일 수정은 작업 브랜치 생성 후 시작한다.
- 작업 완료 후에는 PR로 병합한다.
- 병합 후 사용하지 않는 브랜치는 로컬과 원격에서 정리한다.

## 2.4 병합 규칙

- 일반 작업 브랜치는 `develop` 대상으로 PR을 생성한다.
- 배포 시점에는 `develop`에서 검증된 내용을 `main`으로 올린다.
- 긴급 운영 수정은 `hotfix/*` 브랜치로 처리하고, `main` 병합 후 `develop`에도 반영한다.

## 2.5 커밋 메시지 규칙

커밋 메시지는 아래 형식을 따른다.

`<type>: <summary>`

허용 타입:

- `feat`
- `fix`
- `docs`
- `chore`
- `refactor`
- `style`
- `test`

예시:

- `feat: add dashboard entry cards for presentations`
- `fix: normalize slide paths before routing`
- `docs: define git workflow and branch strategy`

커밋 메시지 규칙:

- 한 줄 요약은 72자 안팎으로 유지한다.
- 요약은 명령문이 아니라 결과 중심으로 작성한다.
- 여러 목적을 한 커밋에 섞지 않는다.

## 2.6 Pull Request 규칙

PR은 아래 내용을 포함하는 것을 기본으로 한다.

- 변경 목적
- 주요 변경점
- 검증 방법
- 배포 영향 여부

PR 제목 역시 커밋 규칙과 유사한 형식을 권장한다.

# 3. 네이밍 규칙

## 3.1 공통 원칙

- 이름만 보고 역할을 추론할 수 있어야 한다.
- 축약어보다 의미 있는 전체 단어를 우선한다.
- 단수/복수는 실제 의미에 맞게 사용한다.

## 3.2 파일명

- 문서 파일: 대문자 고정명 사용
  - 예: `AGENT.md`, `README.md`, `CONVENTION.md`
- 일반 코드 파일: `kebab-case` 또는 프레임워크 관례를 따른다
- 컴포넌트 파일: 프레임워크가 요구하면 `PascalCase` 허용

## 3.3 코드 네이밍

- 변수와 함수: `camelCase`
- 컴포넌트: `PascalCase`
- 상수: `UPPER_SNAKE_CASE`
- 타입과 인터페이스: 프레임워크/언어 관례를 따르되 역할이 드러나게 작성

## 3.4 브랜치와 슬러그 네이밍

- URL, 브랜치, 데이터 식별자에는 공백을 쓰지 않는다.
- 단어 구분은 하이픈을 기본으로 한다.
- 발표 `id`는 사람이 읽을 수 있고 중복되지 않아야 한다.

# 4. 디렉터리 및 파일 구조 규칙

## 4.1 기본 원칙

- 페이지와 공용 컴포넌트를 분리한다.
- 발표 메타데이터와 실제 발표 파일을 분리한다.
- 협업 문서는 `.mc/`에만 둔다.

## 4.2 권장 구조

권장 구조는 아래와 같다.

```text
/
├─ AGENT.md
├─ .mc/
├─ src/
│  ├─ pages/
│  ├─ components/
│  ├─ data/
│  ├─ styles/
│  └─ utils/
├─ public/
│  └─ slides/
└─ ...
```

## 4.3 구조 배치 원칙

- 대시보드 페이지는 목록 탐색 책임만 가진다.
- 발표 페이지는 발표 실행 책임만 가진다.
- 발표 메타데이터는 별도 파일에서 관리한다.
- 실제 발표 HTML은 배포 시 그대로 제공되는 경로에 둔다.

---

# 5. 코드 스타일 규칙

## 5.1 일반 원칙

- 한 파일은 하나의 분명한 책임을 가진다.
- 복잡한 로직은 작은 함수로 나눈다.
- 공통 로직은 반복이 확인될 때만 추출한다.

## 5.2 문자열과 상수

- 반복되는 경로, 라벨, 상태값은 상수화한다.
- 발표 경로 처리 로직에서 문자열 이어붙이기를 남발하지 않는다.

## 5.3 조건문과 흐름 제어

- guard clause를 우선한다.
- 복잡한 분기 로직은 함수로 분리한다.
- 경로 해석, 메타데이터 검증, 라우팅 결정은 섞지 않는다.

## 5.4 주석 규칙

- 주석은 의도를 설명할 때만 사용한다.
- 코드 자체를 번역하는 주석은 쓰지 않는다.
- 오래 유지되지 않을 임시 주석은 남기지 않는다.

## 5.5 에러 처리

- 무시 가능한 에러와 치명적 에러를 구분한다.
- 사용자에게 보여야 하는 상태와 내부 디버깅 상태를 구분한다.
- 실패를 조용히 삼키지 않는다.

# 6. 문서 작성 규칙

## 6.1 Markdown 구조

- 제목 계층은 순서대로 사용한다.
- 섹션은 역할별로 분리한다.
- 규칙 문서는 선언형 문장으로 작성한다.
- 예시는 필요할 때만 넣고, 실제 프로젝트 기준으로 작성한다.

## 6.2 문체 규칙

- 짧고 단정적인 문장 사용
- 모호한 표현 최소화
- "가능하면", "적당히" 같은 표현 남용 금지
- 규칙과 권장사항을 구분해서 작성

## 6.3 문서 동기화

아래 항목이 바뀌면 관련 문서를 함께 갱신한다.

- 브랜치 전략
- 발표 저장 위치
- 메타데이터 구조
- 라우팅 구조
- 배포 절차

# 7. 발표 데이터 규칙

## 7.1 메타데이터 필드 권장안

발표 목록 데이터는 아래 필드를 기본으로 권장한다.

- `id`
- `title`
- `path`
- `updatedAt`
- `description`
- `tags`
- `thumbnail`

## 7.2 필드 설계 원칙

- `id`는 고유해야 한다.
- `title`은 대시보드에서 바로 읽히는 제목이어야 한다.
- `path`는 실제 배포 경로와 일치해야 한다.
- `updatedAt`은 정렬 기준으로 활용 가능해야 한다.
- 선택 필드는 없어도 동작이 깨지지 않아야 한다.

## 7.3 호환성 규칙

- 기존 필드 의미를 바꿀 때는 문서와 데이터를 함께 갱신한다.
- 경로 규칙을 바꿀 때는 기존 링크 호환성을 먼저 검토한다.

# 8. UI/UX 구현 규칙

## 8.1 대시보드

대시보드는 아래 기능을 기준으로 설계한다.

- 발표 목록 확인
- 검색 또는 필터
- 최신 발표 빠른 접근
- 발표 카드 또는 리스트 기반 진입

## 8.2 발표 뷰

발표 뷰는 아래에 집중한다.

- 발표 콘텐츠 렌더링
- 전체화면 진입
- 이전 화면 복귀
- 경로 오류 시 안전한 예외 처리

## 8.3 디자인 원칙

- 발표용 서비스답게 한눈에 탐색 가능해야 한다.
- 장식보다 정보 구조를 우선한다.
- 대시보드와 발표 뷰의 역할을 시각적으로 분리한다.
- 모바일과 데스크톱 모두에서 깨지지 않아야 한다.

---

# 9. 검증 규칙

## 9.1 최소 검증 시나리오

UI 변경 시 아래 시나리오를 우선 확인한다.

1. 루트 진입 시 대시보드가 정상 표시되는가
2. 발표 목록이 정상 렌더링되는가
3. 발표 선택 후 올바른 발표 뷰로 이동하는가
4. 잘못된 경로에 대해 안전하게 대응하는가
5. 뒤로가기와 새로고침이 치명적으로 깨지지 않는가

## 9.2 배포 전 확인 항목

- 상대 경로와 절대 경로 규칙이 일관적인가
- 정적 배포 환경에서 참조 불가능한 경로가 없는가
- 발표 HTML이 실제 배포 경로에서 열리는가

# 10. 최종 원칙

이 문서는 단순 스타일 가이드가 아니라 협업 비용을 줄이기 위한 운영 규약이다.

새 코드나 문서를 만들 때마다 아래를 확인한다.

- 이름이 일관적인가
- 위치가 자연스러운가
- 브랜치 전략과 충돌하지 않는가
- 다음 에이전트가 읽고 바로 이해할 수 있는가
