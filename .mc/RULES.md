# RULES.md

이 문서는 AI 에이전트가 이 프로젝트에서 작업할 때 반드시 따라야 하는 협업 규칙과 안전 기준을 정의한다.
목표는 높은 자율성을 허용하되, 프로젝트의 구조와 운영 안정성을 깨지 않도록 안전선을 명확히 두는 것이다.

이 문서는 구현 방식보다 우선하는 실행 규칙 문서이며, 위험성 판단과 작업 중단 기준의 기준점이 된다.

---

# 1. 기본 원칙

## 1.1 절대 원칙

에이전트는 아래 원칙을 항상 지켜야 한다.

- 사용자가 명시적으로 요청한 범위 안에서만 작업한다.
- 불확실한 내용을 사실처럼 전제하고 구현하지 않는다.
- 정적 프론트엔드라는 시스템 제약을 임의로 깨지 않는다.
- 기존 사용자 작업물과 Git 이력을 존중한다.
- 실패나 제약을 숨기지 않고 명확하게 보고한다.

## 1.2 금지된 태도

아래 태도는 금지한다.

- "아마 이게 더 좋을 것 같다"는 추측 기반 확장
- 요청되지 않은 기능 추가
- 승인 없는 구조 변경
- 배포 환경을 모른 채 설정 수정
- 검증 없이 핵심 흐름 수정

---

# 2. 작업 범위 규칙

## 2.1 기본 허용 범위

아래 작업은 일반적으로 허용된다.

- 문서 작성 및 문서 보강
- UI 구현 및 스타일 수정
- 발표 대시보드와 발표 뷰 관련 프론트엔드 기능 수정
- 정적 메타데이터 구조 보강
- 사용자가 명확히 요청한 버그 수정

## 2.2 승인 없이 해서는 안 되는 작업

아래 작업은 사용자 승인 없이는 수행하지 않는다.

- 배포 브랜치 전략 변경
- Vercel 배포 설정의 의미 있는 변경
- 라우팅 구조의 대규모 변경
- 여러 디렉터리를 동시에 옮기는 구조 개편
- 외부 의존성 추가
- 빌드 도구 교체
- 기존 발표 파일의 삭제 또는 대규모 이동

## 2.3 범위 초과 금지

작업 도중 아래 행동은 금지한다.

- "이왕이니까 같이" 식의 주변 정리
- 직접 관련 없는 리팩터링
- 향후를 가정한 과설계
- 요청되지 않은 상태 관리나 CMS성 기능 추가

필요성이 보여도 먼저 제안하고 승인을 받아야 한다.

---

# 3. 사용자 승인 기준

## 3.1 반드시 승인 받아야 하는 변경

다음 항목은 실행 전에 사용자 확인이 필요하다.

- 브랜치 전략 변경
- `main` 또는 `develop` 운용 방식 변경
- CI/CD 또는 Vercel 설정 변경
- 새로운 프레임워크 또는 라이브러리 도입
- 발표 자료 저장 위치 변경
- 데이터 구조 호환성을 깨는 메타데이터 변경
- 사용자에게 영향을 주는 라우팅 규칙 변경

## 3.2 승인 요청 방식

승인이 필요한 경우 아래 3가지를 포함해 설명한다.

- 변경 범위
- 위험도 또는 영향도
- 대안 또는 유지 가능한 기존 방식

사용자가 명시적으로 승인하기 전에는 진행하지 않는다.

---

# 4. 오류 처리 및 재시도 규칙

## 4.1 기본 절차

오류가 발생하면 아래 순서를 따른다.

1. 재현 가능한지 확인한다.
2. 원인을 좁힌다.
3. 확실한 원인만 수정한다.
4. 같은 방식의 재시도는 한 번만 한다.
5. 재차 실패하면 다른 접근으로 전환한다.

## 4.2 중단 기준

아래 상황이면 작업을 멈추고 사용자에게 상황을 설명한다.

- 동일 원인으로 반복 실패
- 현재 변경이 기존 자료를 손상시킬 가능성 존재
- Git 상태 충돌로 안전한 진행이 어려움
- 배포 영향이 있는데 로컬에서 검증할 수 없음
- 문서 규칙끼리 해석 충돌이 발생

## 4.3 금지된 오류 대응

아래 방식은 허용하지 않는다.

- 에러를 무시하고 넘어가기
- 임시 주석 처리로 숨기기
- 근거 없는 설정 변경으로 우회하기
- 사용자가 모르는 상태에서 실패를 남겨두고 종료하기

---

# 5. Git 및 브랜치 안전 규칙

## 5.1 보호 브랜치 원칙

아래 브랜치는 보호 대상이다.

- `main`
- `develop`

이 브랜치에서는 직접 구현보다 PR 기반 병합을 원칙으로 한다.

## 5.2 금지된 Git 작업

사용자 명시 요청 없이는 아래 작업을 하지 않는다.

- `git reset --hard`
- `git checkout -- <file>`
- 강제 푸시
- 히스토리 재작성
- 무차별 브랜치 삭제

## 5.3 작업 브랜치 규칙

- 작업 시작 전 적절한 브랜치에서 작업 중인지 확인한다.
- 코드 수정은 작업 브랜치에서만 수행한다.
- 병합 완료 후 불필요한 작업 브랜치는 정리한다.

---

# 6. 파일 및 데이터 보호 규칙

## 6.1 발표 파일 보호

이 프로젝트의 핵심 자산은 발표 HTML 파일과 발표 메타데이터다.
따라서 아래를 지켜야 한다.

- 기존 발표 파일을 임의 삭제하지 않는다.
- 발표 파일 경로를 호환성 없이 바꾸지 않는다.
- 메타데이터의 `id`, `path` 같은 식별 필드를 함부로 바꾸지 않는다.

## 6.2 대량 수정 제한

아래 작업은 특히 보수적으로 처리한다.

- `/public/slides` 또는 유사 발표 저장 디렉터리 전체 이동
- 발표 메타데이터 포맷 전환
- URL 규칙을 바꾸는 라우팅 수정

---

# 7. 보안 규칙

## 7.1 절대 금지

아래 내용은 코드와 문서 어디에도 남기지 않는다.

- API 키
- 액세스 토큰
- 개인 계정 정보
- 민감한 내부 URL

## 7.2 정적 사이트 기준 보안

정적 프론트 프로젝트라도 아래는 항상 확인한다.

- 외부 스크립트 출처의 신뢰성
- `iframe` 사용 시 필요한 제한 여부
- 사용자 입력이 DOM에 삽입될 때 XSS 가능성
- 쿼리스트링 기반 발표 선택 시 경로 조작 가능성

## 7.3 불확실할 때

보안상 애매한 경우에는 구현보다 중단과 질문을 우선한다.

---

# 8. 문서 동기화 규칙

다음 변경은 문서 갱신 여부를 반드시 확인한다.

- 브랜치 전략 변경
- 디렉터리 구조 변경
- 발표 등록 방식 변경
- 메타데이터 스키마 변경
- 배포 흐름 변경
- 발표 렌더링 방식 변경

문서가 바뀌어야 하는데 코드만 수정하고 종료하는 것은 금지한다.

---

# 9. 품질 게이트

작업 완료 전에 최소한 아래를 확인한다.

- 요청 범위를 정확히 충족했는가
- 관련 없는 파일이 함께 오염되지 않았는가
- 문서와 결과물이 서로 충돌하지 않는가
- 빌드 또는 실행 검증이 가능한 경우 수행했는가
- 검증하지 못한 부분을 명시했는가

---

# 10. 작업 종료 기준

아래 조건을 만족해야 완료로 본다.

- 요구사항 충족
- 문서 규칙 위반 없음
- 위험한 미해결 상태 없음
- 변경 이유와 결과를 설명 가능
- 후속 조치가 필요하면 명확히 남김

---

# 11. 최종 원칙

에이전트는 이 프로젝트에서 자동화된 실행자이지 독단적 의사결정자가 아니다.

판단은 하되 추측하지 않는다.
구현은 하되 범위를 넘지 않는다.
개선은 하되 승인 없이 확장하지 않는다.
