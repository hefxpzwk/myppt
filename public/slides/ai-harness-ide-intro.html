01 / 08 FY 소개

FY를 제대로 이해한
개발 단계 중간 보고

FY의 목적은 코딩 도구가 아니라 개발 실행 품질을 표준화하는 생산성 인프라를 만드는 것입니다.

지금 단계는 기능 완성보다, 팀이 같은 기준으로 시작하고 개선하는 프레임워크 정착에 있습니다.

현재 상태

Build Mode

핵심 스크립트/골격 완료, 프로젝트 문서 본문 작성 진행 전

02 / 08 핵심 목적

FY 핵심 목적 4가지

  • 실행 표준화: `setup`, `doctor`, `launch/exec`로 시작/점검/실행 절차 통일
  • 문서 기반 개발: 코드보다 문서 컨텍스트를 먼저 로딩해 의사결정 일관성 확보
  • 스킬 온디맨드 확장: 필요한 스킬만 설치해 프로젝트 맞춤형으로 확장
  • 병렬 수정 안정성: 동시 수정 시 충돌 감지/순서 제어/검증 체계 확보
03 / 08 현재 단계

지금 어디까지 왔는가

개발 인프라 단계

FY는 기능 개발보다 생산성 기반을 먼저 깔고 있는 단계

핵심 스크립트 준비

`fy` 실행 진입점과 점검 체계가 이미 구성됨

문서 본문 미작성

`docs-ex`는 대부분 0바이트로, 컨텍스트 확정이 다음 과제

04 / 08 구축 기반

지금까지 세팅된 기반

1. setup

상태 디렉토리와 기본 ignore 초기화

2. doctor

필수 파일, 로그인, 스모크 테스트 점검

3. launch / exec

컨텍스트 주입형 실행 흐름 확보

4. 컨텍스트 병합

global/core + standards + project 문서 합쳐 전달

5. 운영 원칙

작은 변경, 문서 동기화, 오버라이드 우선 규칙 반영

05 / 08 강점

현재 구조의 핵심 강점

시작 품질 균등화

개인 숙련도 편차와 무관하게 개발 시작 품질을 일정하게 유지

문서-코드 정합성

개인 습관이 아니라 프로세스로 정합성을 관리할 수 있음

전략 명확성

Build Mode(초기 완성)와 Refine Mode(병렬 개선) 역할이 분명함

06 / 08 핵심 갭

현재 갭 (중요)

  • `project-context.md` 비어 있음: Goal/Scope/Constraints 합의 부재
  • `project-overrides.md` 비어 있음: 테스트/배포/보안 예외 기준 미정
  • `docs-ex` 대부분 비어 있음: 팀 공유, 온보딩, 발표 근거 문서 부족
07 / 08 우선순위

다음 단계 실행 우선순위

1순위

`project-context` 확정: Goal, Scope, Stakeholders, Constraints

2순위

`project-overrides` 확정: 테스트, 배포, 모니터링, SLA, 보안

3순위

`docs-ex` 핵심 문서 작성: README, ARCHITECTURE, PLANS

4순위

Refine Mode 충돌관리 설계를 실제 구현 항목으로 분해

08 / 08 마무리

FY의 본질은 도구가 아니라 팀 실행 체계입니다

지금은 기능 데모보다 기준 정착이 중요한 시점입니다. 기준 문서만 채워지면, Build Mode에서 Refine Mode로 안정적으로 넘어갈 수 있습니다.

"시작을 표준화하고, 문서로 정렬하고, 병렬 개선을 안전하게 만든다."
