AI Coding Agent
실무 운영 가이드

툴 사용법이 아니라 개발 방식 전환: 지시, 검증, 운영

Core Shift

패러다임 전환부터 이해

오해

AI는 코드 생성기다. 필요한 순간에 한 번 호출하면 된다.

현실

AI는 작업을 수행하는 주니어 개발자다. 지시와 관리가 성능을 결정한다.

핵심: "AI로 코딩"이 아니라 "AI에게 일을 시킨다".

Principles

성공률을 올리는 3원칙

1. 작게 쪼개기

큰 작업 1회보다 작은 작업 다회가 정확도와 복구성 모두 높다.

2. 명확한 지시

목표, 범위, 제약, 완료조건을 숫자와 파일 경로로 고정한다.

3. 컨텍스트 관리

README, AGENTS.md, 로그, 관련 파일만 넣어 모델 혼선을 줄인다.

Workflow

실전 5단계 워크플로우

  • 0단계: 구조 분석만 지시, 코드 작성 금지
  • 1단계: 설계안 먼저 받기 (파일 구조/API 흐름)
  • 2단계: 작업을 기능 단위로 분할 실행
  • 3단계: 테스트/린트/타입체크 자동 검증
  • 4단계: 리뷰 프롬프트로 회귀 위험 점검
먼저 구조를 분석해. 아직 코드는 작성하지 마.
설계안을 확인한 뒤 승인하면 구현을 시작해.
Prompt Design

작업 지시서 템플릿

목표: 로그인 API에 JWT refresh token 추가
범위: auth/*, api/login.ts만 수정
비범위: frontend/*, DB schema 변경 금지
완료조건: 테스트 3개 추가, 기존 테스트 통과, lint 0
실행순서: 코드 수정 -> 테스트 -> 변경 요약
출력형식: 변경 파일 + 변경 이유 + 남은 리스크

팁: "어떻게"보다 "어디까지"를 먼저 고정하면 품질이 안정된다.

CLI Pattern

CLI 기본 사용 루프

codex .
# 또는 claude-code .

read all files and summarize
analyze architecture first, no coding yet
implement task 1 only
run tests and fix failures
summarize changed files
  • 항상 탐색 -> 구현 -> 검증 -> 요약 순서 유지
  • 한 세션에서 한 목표만 다뤄 맥락 오염 방지
Task Decomposition

큰 기능을 쪼개는 예시

나쁜 지시

"인증 기능 전체 만들어줘"

좋은 지시

1) DB schema 2) API route 3) auth middleware 4) 테스트 5) 문서 업데이트

효과: 실패 지점 분리, 롤백 용이, 리뷰 속도 향상.

Role Split

역할 기반 에이전트 운영

Planner

요구사항 해석, 설계안 작성, 작업 분해

설계

Builder

코드 수정, 테스트 실행, 에러 해결

구현

Reviewer

회귀, 보안, 누락 테스트 검토

리뷰

원칙: 구현 프롬프트와 리뷰 프롬프트를 분리한다.

Context Files

컨텍스트 파일 운영

  • README: 프로젝트 개요, 실행 방법, 테스트 커맨드
  • AGENTS.md: 코딩 규칙, 금지사항, 브랜치 전략
  • CONTRIBUTING: PR 기준, 코드 스타일, 리뷰 기준
AGENTS.md 예시
- backend/*만 수정 가능
- 하드코딩 금지
- 예외 처리 필수
- 테스트 실패 시 종료
Tooling

도구별 체감 차이 (실무 관점)

Codex 계열

정교한 수정 루프와 테스트-수정 반복에 강함

Claude Code 계열

긴 문맥 이해, 설계/리팩토링, 대규모 변경 파악에 강함

IDE Agent 계열

인라인 편집과 즉시 피드백이 빠르며 팀 적응이 쉬움

결론: 도구보다 운영 방식이 성과를 좌우한다.

Quality

검증 없는 자동화는 사고다

필수 게이트

  • 테스트 실패 시 작업 중단
  • lint/type error 0 유지
  • 변경 파일별 이유 보고

리뷰 체크

  • 회귀 위험
  • 예외 처리 누락
  • 경계 케이스 테스트
Limits

반드시 알아야 할 한계

  • 장시간 세션에서 맥락 손실이 발생할 수 있다
  • 존재하지 않는 API/옵션을 제시할 수 있다
  • 터미널/환경 의존 이슈에서 실패율이 올라간다
  • 결과를 과신하면 결함이 누적된다

정답: 작은 작업 단위 + 빈번한 검증 + 인간 승인.

Security

보안/정책 운영 기준

  • 민감정보(토큰, 고객 데이터) 프롬프트 입력 금지
  • 허용 디렉토리/명령 화이트리스트 운영
  • 로그 감사 가능하게 지시/변경 이력 보관
  • 외부 모델 데이터 보존 정책 사전 확인
People

잘 쓰는 팀 vs 망하는 팀

망하는 팀

큰 작업 일괄 지시, 모호한 요구, 검증 생략, 방식 제각각

잘 쓰는 팀

작은 단위 지시, 명확한 완료조건, 테스트 자동화, 공통 템플릿

Metrics

성과 측정 지표

  • TTR: 이슈 해결 리드타임
  • FTR: 첫 시도 통과율
  • RR: 회귀율
  • PR 왕복 횟수: 커뮤니케이션/품질 성숙도 지표
  • 작업당 비용: 모델/세션 운영 최적화 지표
Rollout

조직 도입 로드맵

파일럿

2주, 작은 백로그 중심

표준화

템플릿, 규칙, 품질 게이트 확정

확장

CI 연동, 비용/성과 모니터링

핵심: 개인 요령이 아니라 팀 표준 프로세스로 정착.

Ready-to-Use

바로 쓰는 실전 프롬프트

너는 backend engineer야.
목표: 주문 취소 API 오류 수정.
범위: services/order/* 만 수정.
제약: DB schema 변경 금지.
완료조건: 테스트 추가 2개, 전체 테스트 통과, lint 0.
순서: 분석 -> 계획 보고 -> 구현 -> 테스트 -> 요약.
출력: 변경 파일, 이유, 남은 리스크 3개.
Closing

한 줄 결론

AI coding agent는 코드 생성기가 아니라, 관리해야 하는 실행형 팀원이다.

  • 작게 쪼개고
  • 명확하게 지시하고
  • 항상 검증하라
