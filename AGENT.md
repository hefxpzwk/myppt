# AGENT.md

이 문서는 이 프로젝트에서 작업하는 AI 에이전트의 최상위 실행 지침을 정의한다.
모든 에이전트는 이 문서를 시작점으로 사용하며, 실제 구현과 수정은 반드시 `.mc` 문서 체계를 기준으로 수행한다.

이 프로젝트에서 AI는 생산성을 높이기 위한 실행 도구이지만, 독립적인 제품 설계자나 임의 확장 주체가 아니다.
즉, 자유롭게 코드를 작성할 수는 있으나 문서화된 규칙과 아키텍처 바깥으로 벗어나는 행동은 허용되지 않는다.

---

# 1. 문서 우선 원칙

## 1.1 필수 사전 작업

사용자 요청을 받은 에이전트는 작업 전에 반드시 아래 문서를 순서대로 읽는다.

1. `.mc/RULES.md`
2. `.mc/CONVENTION.md`
3. `.mc/ARCHITECTURE.md`

이 순서를 생략하거나 일부만 읽고 작업하는 것은 금지한다.

## 1.2 문서 역할

각 문서의 책임은 다음과 같이 고정한다.

- `.mc/RULES.md`
  - 작업 허용 범위, 승인 기준, 위험 작업 제한, 오류 대응 규칙
- `.mc/CONVENTION.md`
  - 코드 스타일, 문서 스타일, 네이밍, 브랜치, 커밋, PR 규칙
- `.mc/ARCHITECTURE.md`
  - 프로젝트 목적, 기술 구조, 폴더 설계, 데이터 구조, 배포 모델

에이전트는 작업 도중 판단이 필요할 때마다 위 문서를 다시 참조해야 한다.

## 1.2.1 프로젝트 스킬

이 저장소에는 반복 작업을 줄이기 위한 프로젝트 전용 스킬이 `.mc/skills/` 아래에 존재할 수 있다.

- `.mc/skills/presentation-registration/SKILL.md`
  - 발표 등록, 발표 메타데이터 수정, 슬라이드 경로 검증 작업에 사용
- `.mc/skills/presentation-hub-frontend/SKILL.md`
  - 대시보드, 발표 뷰, 정적 프론트엔드 UI 작업에 사용

이 스킬들은 `AGENT.md`와 `.mc` 문서를 보조하는 실행 가이드다.
우선순위를 대체하지 않으며, 필수 문서를 읽은 뒤 작업 유형이 맞을 때만 참조한다.

## 1.3 문서 우선순위

문서 간 판단 우선순위는 아래와 같다.

1. `AGENT.md`
2. `.mc/RULES.md`
3. `.mc/CONVENTION.md`
4. `.mc/ARCHITECTURE.md`
5. 사용자의 현재 요청
6. 기존 코드

단, 안전성과 파괴적 작업 방지에 관한 판단은 언제나 `RULES.md`를 우선 적용한다.

# 2. 프로젝트 이해

## 2.1 프로젝트 목적

이 프로젝트는 여러 HTML 발표 자료를 한곳에서 관리하고 실행하기 위한 정적 프론트엔드 허브를 만든다.

핵심 목적은 다음과 같다.

- 발표 자료를 만들 때마다 개별 프롬프트, 스킬, 임시 설정을 반복하지 않도록 한다.
- 배포된 웹에서 발표 목록을 탐색하고 즉시 발표 모드로 진입할 수 있게 한다.
- 로컬에서 생성한 HTML 발표 파일을 GitHub에 반영하고, Vercel 자동 배포로 운영한다.

## 2.2 핵심 사용자 흐름

이 프로젝트의 기본 사용자 흐름은 아래와 같다.

1. 사용자가 서비스 루트에 접속한다.
2. 대시보드에서 발표 목록을 확인한다.
3. 원하는 발표를 선택한다.
4. 발표 전용 뷰로 진입한다.
5. 전체화면 또는 일반 모드에서 발표를 진행한다.

## 2.3 시스템 제약

에이전트는 아래 전제를 항상 유지해야 한다.

- 서버를 두지 않는다.
- 런타임 백엔드 API를 전제로 설계하지 않는다.
- 발표 자료는 외부 CMS가 아니라 저장소 내부 정적 파일로 관리한다.
- 배포는 Git 기반으로 추적 가능해야 한다.

# 3. 작업 실행 절차

에이전트는 모든 작업을 아래 절차에 따라 수행한다.

## Step 1. 요청 해석

- 사용자의 요청을 한 문장으로 요약한다.
- 요청된 범위와 요청되지 않은 범위를 구분한다.
- 문서 수정인지, 코드 구현인지, 구조 변경인지 먼저 분류한다.

## Step 2. 문서 기준 검증

- 요청이 `RULES.md`의 허용 범위 안에 있는지 확인한다.
- 결과물이 `CONVENTION.md`를 만족하는지 확인한다.
- 구조적 방향이 `ARCHITECTURE.md`와 충돌하지 않는지 검토한다.

## Step 3. 브랜치 상태 확인

- 현재 브랜치를 확인한다.
- `main`에서 직접 작업 중인지 확인한다.
- 코드 변경이 필요한 경우 적절한 작업 브랜치에서 작업 중인지 확인한다.
- 브랜치 규칙은 반드시 `.mc/CONVENTION.md`의 Git 규칙을 따른다.

## Step 4. 작업 범위 확정

- 수정 대상 파일 목록을 먼저 식별한다.
- 관련 없는 파일 수정 가능성을 배제한다.
- 새 파일이 필요하면 생성 위치와 네이밍 규칙을 먼저 결정한다.

## Step 5. 구현 또는 문서 수정

- 최소 변경 원칙으로 작업한다.
- 구조를 바꾸는 변경은 필요한 경우에만 수행한다.
- 문서와 코드가 동시에 바뀌어야 하는 경우 둘 다 반영한다.

## Step 6. 검증

- 문법 오류, 빌드 오류, 명백한 런타임 문제를 점검한다.
- 사용자 흐름 기준 검증이 필요한 경우 핵심 플로우를 확인한다.
- 문서 변경이라면 문서 간 충돌 여부를 확인한다.

## Step 7. 결과 보고

작업 완료 시 아래 항목을 요약한다.

- 무엇을 변경했는지
- 왜 그렇게 변경했는지
- 어떤 검증을 했는지
- 남아 있는 제약 또는 후속 작업이 있는지

# 4. 브랜치 작업 원칙

이 프로젝트는 Git 기반 협업을 전제로 하며, 브랜치 규칙은 문서 수준에서 강제한다.

## 4.1 브랜치 역할

- `main`
  - 배포 기준 브랜치
  - Vercel이 참조하는 운영 브랜치
- `develop`
  - 개발 통합 브랜치
  - 기능 병합과 검토가 먼저 모이는 기본 브랜치

## 4.2 작업 브랜치 원칙

모든 일반 작업은 `develop`에서 분기한 작업 브랜치에서 수행한다.

허용 브랜치 유형 예시:

- `feat/<slug>`
- `fix/<slug>`
- `docs/<slug>`
- `chore/<slug>`
- `refactor/<slug>`

핫픽스가 필요한 경우에만 `main`에서 `hotfix/<slug>`를 분기할 수 있다.

## 4.3 금지 사항

- `main`에서 직접 구현, 직접 커밋, 직접 푸시
- 작업 브랜치 없이 파일 수정 시작
- PR 없이 임의 병합
- 병합 후 사용하지 않는 브랜치 방치
- 브랜치 목적과 맞지 않는 이름 사용

---

# 5. 충돌 처리 규칙

## 5.1 사용자 요청과 문서가 충돌할 때

문서와 충돌하는 요청은 바로 실행하지 않는다.

에이전트는 아래를 명확히 설명해야 한다.

- 어떤 문서와 충돌하는지
- 충돌 이유가 무엇인지
- 가능한 대안이 무엇인지

그 후 사용자 결정을 받아야 한다.

## 5.2 문서와 기존 코드가 충돌할 때

기존 코드가 문서 규칙과 충돌하면 아래 순서로 판단한다.

1. 현 구조가 의도된 예외인지 확인한다.
2. 예외가 아니라면 문서에 맞게 수정 가능한지 검토한다.
3. 영향이 크면 사용자와 합의한 뒤 진행한다.

## 5.3 문서 간 충돌이 발생할 때

- 안전 규칙은 `RULES.md`를 우선한다.
- 작업 방식과 표현 규칙은 `CONVENTION.md`를 따른다.
- 구조 설계는 `ARCHITECTURE.md`를 따른다.
- 그래도 해소되지 않으면 사용자에게 판단을 요청한다.

---

# 6. 절대 금지 행동

에이전트는 아래 행동을 해서는 안 된다.

- `.mc` 문서를 읽지 않고 구현 시작
- 요청되지 않은 기능 추가
- 서버나 백엔드 전제를 임의로 도입
- 비밀 정보나 토큰을 코드/문서에 기록
- 파괴적 Git 작업 수행
- 배포 브랜치에 직접 푸시 전제의 설계 변경
- 발표 자료 관리 흐름을 임의로 CMS/DB 구조로 확장

---

# 7. 문서 동기화 의무

다음 변경이 발생하면 에이전트는 문서 갱신 필요 여부를 반드시 점검한다.

- 폴더 구조 변경
- 라우팅 구조 변경
- 발표 메타데이터 구조 변경
- 배포 방식 변경
- 브랜치/커밋/PR 규칙 변경
- 보안 정책 변경

문서 갱신이 필요하면 코드만 바꾸고 종료해서는 안 된다.

---

# 8. 완료 기준

작업은 아래 조건을 만족해야 완료로 간주한다.

- 사용자 요청 범위를 충족한다.
- `RULES.md` 위반이 없다.
- `CONVENTION.md` 기준을 만족한다.
- `ARCHITECTURE.md`와 정합성이 있다.
- 필요한 문서 반영이 누락되지 않았다.
- 검증 결과 또는 검증 불가 사유를 명확히 보고했다.

---

# 9. 최종 원칙

이 프로젝트에서 에이전트의 자유도는 문서 안에서만 허용된다.

에이전트는 작업마다 아래 질문을 스스로 확인해야 한다.

`이 작업은 문서 기준으로 정당한가?`

정당하지 않다면 수행하지 않는다.
